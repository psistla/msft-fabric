## AI-Ready Data Strategy: Component Analysis

Validated against current Microsoft Fabric capabilities, Gartner research, and 2025–2026 enterprise AI practice. Gartner predicts that through 2026, organizations will abandon 60% of AI projects unsupported by AI-ready data [Gartner](https://www.gartner.com/en/newsroom/press-releases/2025-02-26-lack-of-ai-ready-data-puts-ai-projects-at-risk) — the analysis below is structured to prevent exactly that.

---

| # | Component | Best Practices | Work Required | Risks & Unknowns | Assumptions |
|---|-----------|----------------|---------------|------------------|-------------|
| **01** | **Data Ingestion & Source Connectivity** | (1) Metadata-driven, parameterized pipelines — ForEach over config tables, never one pipeline per table. (2) Use mirroring for zero-ETL connectivity to Azure SQL, Cosmos DB, PostgreSQL, Snowflake, and Oracle — eliminating cumbersome ETL pipelines and keeping data in sync in near real-time. [Microsoft Fabric](https://blog.fabric.microsoft.com/en-US/blog/onelake-your-foundation-for-an-ai-ready-data-estate/) (3) Sensitivity labels applied at landing, not post-hoc — PHI/PCI tagged on arrival before any compute touches the data. | Map all source systems and connectivity methods. Evaluate mirroring vs. pipeline per source. Build parameterized pipeline framework. Configure FHIR connector for EHR sources. Stand up Private Endpoints for all production traffic. | Unknown source system API stability and rate limits. Legacy systems (COBOL, on-prem Oracle) may lack modern connectors and require custom JDBC/ODBC patterns. Schema drift from upstream systems silently breaks downstream. | Assumes source system owners will cooperate with access provisioning. Assumes network connectivity from Azure to on-prem sources is approved and feasible. |
| **02** | **Storage Architecture (OneLake / Medallion)** | (1) Delta Parquet as the universal format across Bronze, Silver, and Gold — open, queryable by every Fabric engine without copy. (2) Ingest data once into OneLake and make it instantly usable across analytics, AI, and applications — shortcuts and mirroring eliminate redundant copies and data silos. [Microsoft Blogs](https://blogs.microsoft.com/blog/2025/09/16/microsoft-leads-shift-beyond-data-unification-to-organization-delivering-next-gen-ai-readiness-with-new-microsoft-fabric-capabilities/) (3) Apply V-ORDER write optimization on all Gold Delta tables — mandatory for DirectLake semantic model query performance. | Define workspace hierarchy and domain structure. Provision F64+ capacity per environment (Dev/Test/Prod). Implement OPTIMIZE and VACUUM schedules on 7-day cadence. Configure OneLake Access Policies for domain isolation. | Fabric capacity right-sizing is unknown until actual workload patterns emerge. Concurrent AI and analytical workloads compete for CU capacity — sizing a single SKU for both is non-trivial. OneLake shortcut governance gaps between domains. | Assumes commercial Azure, single tenant. Assumes all structured data can land in Delta Parquet. Assumes F64 is the minimum viable production SKU. |
| **03** | **Data Classification & Sensitivity Labeling** | (1) Microsoft Purview auto-classification using 200+ built-in sensitive information types — SSN, MRN, credit card, IBAN, DOB — no manual tagging for standard patterns. (2) Four-tier label taxonomy: Public → Internal → Confidential (CUI/PHI) → Restricted — applied at ingestion, inherited by all derived assets. (3) Purview DLP automatically scans for regulated information types such as patient IDs, SSNs, medical diagnoses, and insurance data across Lakehouses and clinical models, blocking unsafe uploads in real time. [Microsoft Fabric](https://blog.fabric.microsoft.com/en-us/blog/meet-your-healthcare-regulation-and-compliance-requirements-with-purview-data-loss-prevention-dlp-policies/) | Run baseline Purview scan across all registered data sources. Define label taxonomy aligned to HIPAA/PCI classification tiers. Configure DLP policies per classification tier. Test label inheritance through Silver and Gold layers. Verify labels propagate to Power BI exports and AI Foundry retrieval. | Custom data types (internal codes, proprietary identifiers) won't match built-in classifiers — require custom regex patterns. Unstructured data (PDFs, clinical notes) classification accuracy varies. False positive rate on DLP policies may disrupt legitimate workflows. | Assumes data domain owners validate classification accuracy. Assumes Purview scan has connectivity to all source systems. Assumes label taxonomy is agreed before ingestion begins. |
| **04** | **Data Quality Management** | (1) DQ rules enforced at the Silver transformation layer using PySpark assertion notebooks — completeness, accuracy, timeliness, consistency, uniqueness — with automated pipeline failure on threshold breach. (2) No Gold promotion without a composite DQ score ≥95% — enforced by CI/CD gate, not manual review. (3) Address quality early — key attributes include relevance, accuracy, completeness, consistency, and diversity to reduce bias — because with GenAI, data quality issues multiply, not just propagate. [World Economic Forum](https://www.weforum.org/stories/2026/01/why-data-readiness-is-now-a-strategic-imperative-for-businesses/) | Define DQ dimensions and thresholds per data domain. Build assertion notebooks per domain. Wire assertions into CI/CD pipeline gates. Build DQ dashboard in Power BI. Establish escalation path for failed domain promotions. | Thresholds are unknown until baseline quality is measured — assume they will be worse than expected. DQ for unstructured data (text, documents) requires different tooling than structured tables. Historical data quality debt in existing systems is unquantified. | Assumes DQ rules are agreed with Data Domain Owners before transformation build. Assumes 95% is the right threshold — may need to be domain-specific. |
| **05** | **Data Governance & Catalog** | (1) Every Gold dataset registered as a formal data product in the Purview catalog with owner, SLA, classification, lineage, and AI eligibility flag — not optional. (2) Business glossary with canonical term definitions — "patient," "member," "account" — before any semantic model is built. (3) Microsoft Purview and MDM shift from supporting tools to core architectural components — governance cannot be bolted on later when AI is interpreting and summarizing data that executives act on. [DynaTech Consultancy](https://dynatechconsultancy.com/blog/ai-ready-data-governance-microsoft-fabric-purview-mdm) | Register all active data sources in Purview catalog. Define business glossary for top 30 business terms per domain. Assign Data Domain Owners and Data Stewards. Implement Purview Access Policies for OneLake domain isolation. Establish quarterly access review cadence via Entra ID. | Data ownership is politically contested in most organizations — agreeing owners per domain is harder than the tooling. Business glossary completeness is a continuous effort, not a project. Lineage gaps where non-Fabric tools are in the pipeline (e.g., external ETL) will show breaks. | Assumes executive sponsorship to enforce data ownership decisions. Assumes the Purview catalog is the single authoritative catalog — no competing tools. |
| **06** | **Identity, Access & Security** | (1) Zero Trust identity plane — Entra ID RBAC at workspace level, RLS at semantic model layer, CLS at column level, Purview Access Policies at OneLake storage layer — defense in depth across all four tiers. (2) PIM (Privileged Identity Management) for just-in-time access to PHI/PCI tables — no standing access, time-limited activation, approval required, fully logged. (3) CMK (Customer-Managed Keys) via Azure Key Vault HSM for Highly Confidential and Restricted workspaces — automated 90-day key rotation. | Configure Entra ID Conditional Access policies (MFA, compliant device). Implement PIM for sensitive workspace roles. Deploy CMK for PHI-containing workspaces. Define RLS roles in all Gold semantic models. Test access isolation — verify User A cannot see User B's restricted rows. Wire all secrets to Key Vault — zero credentials in code. | Key rotation operational overhead is non-trivial at scale. RLS complexity grows non-linearly with number of roles and data domains. External vendor or partner access patterns are unknown and may require B2B guest account governance. | Assumes all users are in the same Entra ID tenant. Assumes MFA is enforceable across all user populations including clinical staff. Assumes no legacy service accounts require standing high-privilege access. |
| **07** | **Semantic Layer & Reporting** | (1) One DirectLake semantic model per domain — single definition of every KPI, enforced at model level, never duplicated in reports. (2) Use Power BI's "Prep for AI" feature — AI data schemas, table/column descriptions, and example queries — because the DAX generation tool used by Fabric data agents relies solely on semantic model metadata; proper Prep for AI configuration is essential for accurate AI responses. [Microsoft Learn](https://learn.microsoft.com/en-us/fabric/data-science/semantic-model-best-practices) (3) Sensitivity labels propagate from semantic model to every report and export — no unlabeled PHI-adjacent export is possible. | Build DirectLake semantic models on Gold Delta tables. Configure Prep for AI metadata: table/column descriptions, business glossary terms, example queries, AI data schema scoping. Implement RLS + CLS per role. Section 508 / accessibility validation on all dashboards. | Semantic model complexity grows with domain breadth. DirectLake has query limitations vs. import mode for complex DAX — requires testing against actual analytical patterns. Prep for AI quality directly determines Copilot and Data Agent response accuracy — poor metadata = poor AI answers. | Assumes one model per domain is achievable — some organizations have hundreds of competing models. Assumes Power BI Premium or Fabric F-SKU licensing is in scope. |
| **08** | **Vector Store & Embedding Pipeline** | (1) For advanced AI capabilities such as semantic search or RAG, it is not enough to simply store raw text or images — assets must be chunked and converted into vector embeddings stored in a vector database, enabling efficient similarity search and retrieval. [GOV.UK](https://www.gov.uk/government/publications/making-government-datasets-ready-for-ai/guidelines-and-best-practices-for-making-government-datasets-ready-for-ai) (2) Hybrid retrieval as the default — combine BM25 keyword search with vector semantic search, then apply a reranker — consistently outperforms single-method retrieval on noisy enterprise data. (3) Embeddings drift as chunking strategies shift and embedding models update — this creates silent performance degradation often misdiagnosed as hallucinations; monitoring embedding quality (dimensionality, consistency, vector completeness) is mission-critical. [Towards Data Science](https://towardsdatascience.com/10-data-ai-observations-to-watch-in-fall-2025/) | Choose embedding model aligned to data modality (text, multimodal). Define chunking strategy per document type — semantic/heading-aware chunking preferred over fixed-size. Deploy Azure AI Foundry IQ (AI Search) as the index layer on OneLake. Implement hybrid retrieval + reranking pipeline. Build embedding drift monitoring. | Optimal chunk size is unknown without experimentation — there is a structural tension between recall (small chunks) and context (large chunks). Embedding model selection has long-term lock-in risk. Reindex cost when embedding model changes is significant at scale. | Assumes unstructured source documents are in extractable format (not scanned image PDFs). Assumes English-primary corpus — multilingual data requires different embedding models. Assumes OneLake is the knowledge store — no external data sources in scope. |
| **09** | **RAG & AI Retrieval Pipeline** | (1) Purview sensitivity labels enforced at index time — documents labeled Restricted never surface to users without that clearance, regardless of query. (2) Zero external grounding by default — all retrieval from agency/company-owned OneLake knowledge base; no Bing/public internet grounding. (3) Curate the corpus before indexing — de-duplicate, version, and label documents; add metadata for owner, sensitivity, and effective date — because retrieval quality is determined upstream of the vector database, not within it. [Data Nucleus](https://datanucleus.dev/rag-and-agentic-ai/what-is-rag-enterprise-guide-2025) | Build ingestion pipeline from OneLake Files to AI Foundry IQ index. Implement incremental index refresh on Delta table change — not full re-index on timer. Configure Purview label enforcement at retrieval layer. Build source attribution into every AI response for audit defensibility. Implement prompt injection defense via Purview DLP on input validation. | RAG retrieval quality is non-deterministic — the same query can return different results. Prompt injection is an active threat vector in regulated environments. Index freshness SLA vs. reindex cost is an unresolved tension. Performance degrades with corpus size — requires ongoing tuning. | Assumes documents are owned and curated by the organization — no third-party content licensing issues. Assumes AI Foundry is the retrieval engine — not a third-party RAG framework. |
| **10** | **AI Model Lifecycle & MLOps** | (1) Every model registered in Azure ML Model Registry with version, training data lineage, bias assessment, intended use, and decommission plan — required before production deployment. (2) AI-ready data is not "one and done" — develop an AI-ready data practice with constant improvement based on existing and upcoming AI use cases, including DataOps and data observability to track data patterns and changes. [Gartner](https://www.gartner.com/en/newsroom/press-releases/2025-02-26-lack-of-ai-ready-data-puts-ai-projects-at-risk) (3) Human-in-the-loop gates for all agentic AI decisions above a defined risk threshold — required for regulatory defensibility if an automated decision is challenged. | Build model training pipelines on governed Gold data — never Bronze. Implement SHAP explainability for all predictive models. Configure automated retraining triggers on data drift detection. Build model performance monitoring dashboards. Complete red team assessment before each production deployment. | Model drift detection thresholds are unknown until production baselines are established. Retraining frequency vs. cost vs. stability is a continuous tuning problem. Explainability methods (SHAP) add latency — real-time scoring trade-off is unresolved. | Assumes Azure ML is the model training platform. Assumes data scientists have Fabric Data Science workspace access. Assumes compliance team will review model cards before go-live. |
| **11** | **Real-Time Data & Streaming** | (1) Fabric Event Streams as the ingestion layer for real-time sources — transaction feeds, ADT messages, IoT, clickstreams — with no-code connectors converging into a governed Real-Time hub. (2) KQL databases (Eventhouses) for time-series and high-throughput analytical queries — orders of magnitude faster than SQL for streaming data patterns. (3) Data Activator for automated real-time response — fraud holds, clinical alerts, threshold triggers — sub-second latency from event to action without custom application code. | Define real-time vs. batch boundary per data domain. Build Event Streams topology for high-priority streaming sources (transactions, alerts). Design KQL schemas for time-series analytical patterns. Configure Data Activator rules for automated response triggers. | Streaming data schema evolution is harder to govern than batch — breaking changes arrive at ingestion speed. KQL is a distinct query language — team upskilling required. Real-time SLA vs. cost is an unresolved tension at scale. | Assumes real-time sources have stable event schemas. Assumes latency SLA for fraud detection is sub-second — if 5 minutes is acceptable, batch is sufficient and simpler. Assumes Event Streams has connectivity to all real-time source systems. |
| **12** | **Data Observability & Monitoring** | (1) Shift from static data governance to continuous data quality monitoring with strong metadata and lineage tracking — AI requires far more dynamic, granular, and real-time data pipelines than traditional BI, and retention policies must reflect AI's blend of ephemeral, cached, and saved data. [CIO](https://www.cio.com/article/4104444/8-tips-for-rebuilding-an-ai-ready-data-strategy.html) (2) Pipeline health dashboards — failed runs, SLA breaches, DQ score trends — surfaced in Power BI with Sentinel integration for security event correlation. (3) Monitor unstructured data quality alongside structured — text files, documents, and embeddings need automated quality checks, not just database tables. | Build unified monitoring lakehouse aggregating pipeline run metadata, DQ scores, audit events, and capacity metrics. Build operational dashboards for Data Engineering, ISSO, and Domain Owners. Configure Sentinel alerts for Tier 1 pipeline failures and anomalous access patterns. Implement embedding drift monitoring for all production RAG indexes. | Unstructured data observability tooling is immature — most current tools only handle structured tables. Embedding drift is currently a manual diagnostic process for most teams. Alert fatigue from over-sensitive monitoring will cause teams to ignore warnings. | Assumes Sentinel is the central SIEM — no competing SIEM in scope. Assumes pipeline metadata is captured in Fabric Monitoring Hub — no external observability tool required. |
| **13** | **Responsible AI & Compliance Controls** | (1) Bias and representativeness review mandatory before any model trains on PHI or PCI-adjacent data — documented in model card, reviewed by ISSO. (2) Content Safety filters on all Azure OpenAI endpoints — required for any citizen- or employee-facing AI surface. (3) AI use is only safe if it's secure — organizations must maintain existing security and access constraints so people don't see the data they shouldn't see, even through AI interfaces like Copilot and data agents. [CohnReznick](https://www.cohnreznick.com/insights/microsoft-fabric-2025-new-features-and-key-takeaways) | Draft Responsible AI policy aligned to EO 14110 / EU AI Act principles. Build model card template and enforce completion before production sign-off. Configure Azure AI Content Safety on all production endpoints. Register all AI agents in Entra Agent ID for centralized inventory. Implement Purview "Risky Agents" anomaly detection for Copilot Studio and Foundry agents. | Regulatory landscape is evolving — EU AI Act classification of "high-risk" AI in healthcare/financial services is still being interpreted in practice. Responsible AI review adds time — unknown how long compliance team review cycles will take. Hallucination rate acceptable thresholds for regulated outputs are undefined. | Assumes Responsible AI review is an internal process — no external regulator sign-off required before launch. Assumes Content Safety default filters are sufficient — custom filter rules may be needed for domain-specific harmful content. |
| **14** | **Platform Capacity & Cost Governance** | (1) F64 as the minimum production SKU for concurrent analytical and AI workloads — F32 is insufficient once DirectLake semantic models and AI Foundry RAG pipelines run simultaneously. (2) Non-production capacity on automated pause schedule — off outside business hours — delivers 40–60% cost reduction with no reliability impact.
